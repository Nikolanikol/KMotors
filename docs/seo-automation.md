# SEO-автоматика (kmotors.shop)

Самоподдерживающаяся петля: находим слабые страницы → генерим улучшения (LLM) →
апрувим в Telegram → публикуем и пингуем поисковики. Цель — **запчасти как точка
входа клиента**; каталог авто вторичен (и так частично в топе).

> Приоритет языков: **ru + en**. ar/ka вторичны. ko отключён (каталог закрыт корейцам).

---

## Как это работает (петля)

```
[1 СБОР]      GSC API ──► seo_page_stats            (что показывается в поиске)
[2 ДЕТЕКТ]    SQL-правила ──► список слабых страниц
[3 ГЕНЕРАЦИЯ] LLM (Gemini) ──► seo_suggestions (draft)   ← НИЧЕГО на сайт не идёт
[4 ГЕЙТ]      Telegram дайджест ──► ты жмёшь «Одобрить»
[5 ПУБЛИКАЦИЯ] draft→parts_products ─► revalidate ─► IndexNow (Bing+Яндекс)
```

Ключевой принцип: **между генерацией и сайтом всегда стоит твой апрув.** Пока не
одобрил — на живых страницах ноль изменений.

---

## Хранилища данных

| Таблица / поля | Роль |
|---|---|
| `seo_page_stats` | снапшоты Search Console по страницам (скользящее окно 45 дней) |
| `seo_suggestions` | очередь черновиков LLM со статусом `draft→approved/rejected/applied` + снимок метрик «до» |
| `parts_products.seo_*` | боевые поля (`seo_title/desc/body_ru/en`, `cross_refs`, `seo_updated_at`, `seo_content_hash`) — заполняются только после апрува |

Долгую историю «до/после» хранит `seo_suggestions` (снимок на момент правки), а не
`seo_page_stats` — поэтому статы можно чистить.

## Переменные окружения

```
GSC_SA_JSON        — JSON-ключ service account (одной строкой)
GSC_SITE_URL       — https://www.kmotors.shop/   (URL-prefix ресурс)
SEO_CRON_SECRET    — секрет для /api/seo/* (заголовок x-seo-secret)
GEMINI_API_KEY     — ключ Gemini
LLM_PROVIDER       — "gemini" (по умолчанию)
GEMINI_MODEL       — "gemini-2.5-flash" (по умолчанию)
```

## Запуск вручную

```bash
# сбор данных
curl -X POST -H "x-seo-secret: $SEO_CRON_SECRET" https://www.kmotors.shop/api/seo/collect
# генерация N черновиков
curl -X POST -H "x-seo-secret: $SEO_CRON_SECRET" "https://www.kmotors.shop/api/seo/generate?limit=25"
```

## Гибкость LLM (важно)

Вся работа с LLM спрятана за `src/lib/llm.ts` → интерфейс `LlmClient.generateJSON()`.
Сменить модель/тариф/провайдера = одна env-переменная, код генератора не трогается:

- другой tier/ключ Gemini → просто поменять `GEMINI_API_KEY` / `GEMINI_MODEL`;
- другой провайдер (OpenAI/Anthropic) → добавить класс, реализующий `LlmClient`, и
  ветку в `getLlm()`; выбирается через `LLM_PROVIDER`.

`llm.ts` сам ретраит транзиентные сбои (обрезанный JSON) и кидает `QuotaError` при
исчерпании лимита, чтобы генератор корректно остановил батч.

## ⚠️ Текущее ограничение: Gemini free tier

Ключ на **бесплатном тарифе — 20 генераций/сутки** на `gemini-2.5-flash`
(лимит общий с генератором блога). Поэтому:

- генератор берёт малые батчи, троттлит (пауза `THROTTLE_MS`) и **останавливается на
  квоте** (`quotaHit: true`), не жгёт заведомо провальные запросы;
- пилот на 724 карточки при 20/сутки растянется на ~месяц.

**Чтобы ускорить** — включить биллинг в Google AI Studio (flash копеечный: пилот <$1,
весь каталог 50k — несколько $). Технически меняется только тариф ключа, код готов.

## Мультиплатформенность (будущее)

Есть второй проект с запчастями и схожей архитектурой. План — объединить данные и
запускать механику на две платформы. Что уже сделано в эту сторону:

- LLM за абстракцией (`llm.ts`) — переносится как есть;
- логика детектора/генератора не завязана на конкретный сайт (работает от данных в БД).

Что обсудить при объединении: общий источник данных о деталях (единый каталог/номера),
параметризация `GSC_SITE_URL` и базового URL под каждую платформу, общий или раздельный
пул черновиков `seo_suggestions`.
